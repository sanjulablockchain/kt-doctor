# Dark Mode Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add an opt-in dark mode (System / Light / Dark, header toggle, localStorage-persisted) with zero change to the current light appearance.

**Architecture:** All theming flows through the existing CSS custom properties in `app/globals.css`. Each `--color-*` / `--shadow-*` token is defined once with `light-dark(<light>, <dark>)`; the active theme is chosen by `color-scheme` (`:root` = `light dark` to follow the OS, `[data-theme="light"|"dark"]` forces one). Because `@theme inline` already points every Tailwind color utility at those vars, the whole UI flips with almost no utility-class edits. A client `ThemeProvider` writes `localStorage` + the `data-theme` attribute; a blocking inline script applies it before first paint. Literal `bg-white` becomes a themeable `bg-surface`. The fixed-navy footer/CTA are pinned to light-palette accent values via `[data-on-navy]` so they render identically in both themes.

**Tech Stack:** Next.js 16.2.10 (App Router), React 19, Tailwind CSS v4 (CSS-first `@theme`), next-intl, Vitest + Testing Library, Playwright (MCP) for visual/contrast verification.

## Global Constraints

- **No refactor.** The only class edits allowed are theming-driven token swaps (`bg-white`→`bg-surface`, `ring-white`→`ring-surface`, the header hairline shadow) and adding the toggle. No layout/spacing/structure/API changes.

- **WCAG AA in dark mode:** ≥4.5:1 for normal text, ≥3:1 for large/bold text and non-text UI (borders, icon glyphs, focus rings). Dark-mode brand-button contrast must be **no worse than** the existing light-mode equivalent.

- [ ] **Step 3: Write the implementation**

Create `components/ThemeProvider.tsx`:

```tsx
"use client";

import {
  createContext,
  useCallback,
  useContext,
  useEffect,
  useState,
} from "react";

export type ThemePreference = "system" | "light" | "dark";

type ThemeContextValue = {
  preference: ThemePreference;
  setPreference: (preference: ThemePreference) => void;
};

const STORAGE_KEY = "theme";

// A safe default so components (e.g. Header in isolation tests) can call
// useTheme() without an explicit provider. The real provider overrides it.
const ThemeContext = createContext<ThemeContextValue>({
  preference: "system",
  setPreference: () => {},
});

function readStoredPreference(): ThemePreference {
  try {
    const stored = localStorage.getItem(STORAGE_KEY);
    if (stored === "light" || stored === "dark") return stored;
  } catch {
    // localStorage can throw (privacy mode, disabled). Fall back to system.
  }
  return "system";
}

function applyPreference(preference: ThemePreference) {
  const root = document.documentElement;
  if (preference === "system") {
    root.removeAttribute("data-theme");
  } else {
    root.setAttribute("data-theme", preference);
  }
}

export function ThemeProvider({ children }: { children: React.ReactNode }) {
  // Deterministic initial value so the server render and the first client
  // render match (avoids hydration mismatch). The pre-paint inline script has
  // already applied the correct data-theme attribute; we reconcile React state
  // to it after mount.
  const [preference, setPreferenceState] = useState<ThemePreference>("system");

  useEffect(() => {
    setPreferenceState(readStoredPreference());
  }, []);

  const setPreference = useCallback((next: ThemePreference) => {
    setPreferenceState(next);
    try {
      if (next === "system") localStorage.removeItem(STORAGE_KEY);
      else localStorage.setItem(STORAGE_KEY, next);
    } catch {
      // Ignore storage failures; the in-memory state and DOM still update.
    }
    applyPreference(next);
  }, []);

  return (
    <ThemeContext.Provider value={{ preference, setPreference }}>
      {children}
    </ThemeContext.Provider>
  );
}

export function useTheme(): ThemeContextValue {
  return useContext(ThemeContext);
}
```

## Notes on decisions (for the implementer)

- **Why `data-theme` + `light-dark()`:** it cleanly expresses three states — absent = system (`:root`'s `color-scheme: light dark` follows the OS, and `light-dark()` resolves accordingly), `light`/`dark` = explicit override (sets `color-scheme`). No JS resolves "system" and the provider never needs a `matchMedia` subscription — live OS changes are handled purely by CSS. Each token is defined once (no duplicated dark block).
- **Why deterministic `"system"` initial state:** guarantees server render == first client render (no hydration mismatch). The real applied theme is set pre-paint by the inline script, so there is no visual flash even though React state reconciles a tick later.
- **Why `useTheme` has a default context (no throw):** lets `Header` and other components render in isolated unit tests without wrapping every test in a provider. The real provider always overrides it in the app.
- **Do not** add a `viewport`/`generateViewport` export for `colorScheme` — a static value can't match a client override and a dynamic one forces these statically-generated pages to render at request time. `color-scheme` is set in CSS per `data-theme` instead.
