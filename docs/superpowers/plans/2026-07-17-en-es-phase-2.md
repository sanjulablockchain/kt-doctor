# EN/ES Phase 2 (Services + Blog Content) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Translate the last remaining English-only content on the site: all 21 real service names/descriptions/long-descriptions plus their 5 category names in `data/services.ts`, and all 4 full blog articles (title, excerpt, every section) in `data/stories.ts`. This closes out the EN/ES translation work entirely — after this plan, the whole site is fully bilingual.

**Architecture:** Same `*Es` field pattern used throughout Plan B — add parallel `nameEs`/`descriptionEs`/`longDescriptionEs` (services) and `titleEs`/`excerptEs`/`headingEs`/`bodyEs` (stories) fields directly on each data item. Client components (`ServiceCard`, `ServicesPageContent`, `BlogPageContent`, the homepage's Services pill-cloud and Featured Stories sections) read the active locale via `useLocale()` from `next-intl`, exactly like `NetworkCard`/`ProgramCard`/`ResourceCard` already do. The two Server Component detail pages (`ServiceDetailPage`, `BlogPostPage`) are async and read locale directly from `params` (Next.js merges the route's `[locale]` segment into every leaf page's `params`) instead of calling `getLocale()` from `next-intl/server` — that API throws `"getLocale is not supported in Client Components"` under this project's Vitest setup, so it must not be used.

**Tech Stack:** Next.js (App Router) under `app/[locale]/`, TypeScript, Tailwind CSS, Vitest + React Testing Library, next-intl — same as the rest of the app.

## Global Constraints

- No git repository is in use — do not run `git init` or commit. Every task ends with a manual verification step instead of a commit step.
- Do not use the em dash ("—") in any new copy, English or Spanish.
- Translations must be natural, accurate Spanish, not literal word-for-word — matching the quality already used throughout Plan A and Plan B.
- Per an explicit client decision: service/category **names** get translated (not just descriptions) — a Spanish visitor sees "Examen de Niño Sano" instead of "Well Child Exam" everywhere the service appears (pill cloud, cards, category headings, detail page titles).
- Per an explicit client decision: blog article **dates and author names stay unchanged** in both languages (dates aren't language-specific; author names are proper nouns). Only the title, excerpt, and every section heading/body get translated.
- This project uses `localePrefix: "as-needed"` routing — the default locale (`en`) has no URL prefix, but `es` gets a `/es` prefix on every internal link. Any test asserting an internal `href` under the `es` locale must expect the `/es`-prefixed path (e.g. `/es/services`, not `/services`) — this was the source of a real bug caught in the previous plan; don't repeat it here.
- `data/services.ts` and `data/stories.ts` are the only "Phase 2" data files — no other data file should be touched by this plan.
- Every new test rendering a component that uses `next-intl`'s `Link` or `useTranslations`/`useLocale` needs `renderWithIntl` (from `@/lib/test-utils`), not plain `render` from `@testing-library/react`.
- Mobile responsive throughout, matching the existing design system.
